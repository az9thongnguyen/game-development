# 88 · Studio Sheet Export — closing the animation loop

> Code: `src/games/studio/sheet.{hpp,cpp}` (lib `studio_core`), `studio_scene.cpp` · shared: `anim::frames_in_sheet`
> Tests: `tests/test_studio.cpp` (`test_make_sheet`), `tests/test_flipbook.cpp` (`frames_in_sheet`)
> Seen live: `./build/demo --studio` → **Export Sheet** → then `./build/demo --anim` / `--sandbox`

## Why this chapter exists

Two chapters ago the sandbox learned to play sprite-sheet animations — but the only
sheet in the whole project was `spin_8.hrt`, hand-generated by a throwaway script.
The content tool (the Texture Lab) could make *textures* but not *animations*. This
chapter closes that gap: the Lab can now **export an animated sheet from any recipe**,
saved where the players discover it. After this, the loop is self-hosting — author
in the studio, play in `--anim` or `--sandbox`, with no script and no code in
between. That's the Track-B ("mini studio") promise finally standing on its own.

## Animation for free: scroll a tileable texture

The Lab's textures have one property that turns out to be a gift: they are
**seamlessly tileable** (ch. on the Texture Lab — every basis is periodic, so the
left edge continues into the right). That means you can *scroll* a texture and it
wraps with no seam. Scroll it a little more each frame and you get a smooth, looping
"flow" — an animation — **without generating anything new**:

```cpp
gfx::Image make_sheet(const TextureParams& p, int frames) {
    gfx::Image base = generate(p);                 // one tileable texture
    if (frames <= 1 || base.w <= 0) return base;
    const int size = base.w;
    gfx::Image sheet; sheet.w = size; sheet.h = size * frames;
    sheet.pixels.resize(std::size_t(sheet.w) * sheet.h);
    for (int f = 0; f < frames; ++f) {
        const int shift = (f * size) / frames;     // this frame's scroll offset
        for (int y = 0; y < size; ++y) {
            const gfx::Color* srow = &base.pixels[std::size_t(y) * size];
            gfx::Color*       drow = &sheet.pixels[(std::size_t(f) * size + y) * size];
            for (int x = 0; x < size; ++x)
                drow[x] = srow[(x + shift) % size]; // %size wraps seamlessly
        }
    }
    return sheet;
}
```

The `% size` is the whole trick — it's what makes frame `N-1` flow back into frame
`0` without a jump, so the loop is perfect. No new generator, no new parameters; the
tileability we already had does all the work. `make_sheet(p, 1)` returns the plain
texture, so "1 frame" degrades to exactly the old behaviour.

## Self-describing length: frames = height ÷ width

A sheet has to tell its players how many frames it holds. The lazy options —
a filename convention (`flow_8.hrt`) or a sidecar file — both add parsing and a way
to be wrong. Instead the sheet describes itself by its **shape**. Frames are square
and stacked vertically, so an `N`-frame sheet is `w × (N·w)`, and therefore:

```cpp
int frames_in_sheet(int w, int h) {
    if (w > 0 && h > w && h % w == 0) return h / w;   // tall exact multiple → N frames
    return 1;                                         // anything else → static
}
```

`spin_8.hrt` (48×384) gives 8; a Studio sheet at size 32 with 16 frames (32×512)
gives 16; a plain 32×32 texture gives 1. This one pure function is the *entire*
contract between the exporter and both consumers. The end-to-end smoke test proves
the chain: `make_sheet(p, N)` → `encode_hrt` → `decode_hrt` → `frames_in_sheet` == N,
for N ∈ {4, 8, 16}.

## The two consumers now discover Studio output

Because the frame count is self-describing, wiring the consumers is almost nothing:

- **`--sandbox`** drops its hard-coded `sheet_frames_["spin_8"]=8` registry. It
  loads `sprites/spin_8.hrt` *and* probes `sprites/sheet_00..07.hrt` (Studio slots),
  and when you cycle an actor's **Tex** onto any of them it derives the frame count
  from the image's shape: `s->frames = anim::frames_in_sheet(img.w, img.h)`. Pick a
  Studio sheet, it animates; pick a flat texture, it's static.
- **`--anim`** prefers `sprites/sheet_00.hrt` (the latest export) and falls back to
  the built-in spinner, then derives `frames_` from the shape instead of assuming 8.

Export a sheet in the Lab, switch to either scene, and it's there — the pipeline is
closed.

## In the Lab

Two controls join the Texture Lab panel: a **sheet: N frames** cycle (4 / 8 / 16)
and an **Export Sheet** button. Export runs the current recipe through `make_sheet`
and writes `sprites/sheet_NN.hrt`:

```cpp
void StudioScene::export_sheet() {
    sync_params();
    const gfx::Image sheet = make_sheet(params_, kSheetFrames[sheet_idx_]);
    char name[32]; std::snprintf(name, sizeof(name), "sheet_%02d", sheet_counter_++);
    assets::write_file(std::string("sprites/") + name + ".hrt", gfx::encode_hrt(sheet));
}
```

No sidecar is written — the shape carries the frame count, so a saved sheet needs
nothing else to replay.

## What was deliberately left out

- **Re-editable sheet recipes.** A texture saves a `.recipe` sidecar so it can be
  reloaded into the sliders; a sheet doesn't (yet). The recipe format has no
  `frames` field, and adding one is only worth it once you want to *tweak* an
  exported sheet rather than re-export from the live recipe. YAGNI for now.
- **Non-scroll animations** (pulsing, palette-cycling, rotation) — scrolling is the
  one that falls out of tileability for free; the others need real per-frame
  generation and are their own slice.
- **Vertical/diagonal scroll direction** — horizontal reads clearly; a direction
  control is a slider away when a texture wants it.
